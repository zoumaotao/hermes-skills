# 代码改动决策框架 — 先做取舍，再写代码

> 适用场景：你需要在现有系统上做增强（而非重写），在写代码之前先进行结构化的取舍分析。
>
> 触发条件：用户提出「能不能让 X 更好用 / 更智能 / 加个什么功能」，且 X 已有完整实现，只需增强。

## 核心原则

### 铁律

1. **不改存储格式** — 现有格式能用就不要升级（文本 § 分隔就不改 JSON Lines）
2. **不改工具/API schema** — 不新增参数或 action，所有过滤/增强在渲染层自动完成
3. **零外部依赖** — 不引入新数据库、新服务、新 SDK 除非有压倒性理由
4. **磁盘全量保留，注入按需截断** — 存的时候全存，用的时候只挑最好的用

### 取舍清单（逐一过）

在写出任何代码之前，依次回答以下问题。把答案写进最终 plan 的「被否决方案」部分。

| # | 取舍问题 | 默认答案 | 什么情况下才破例 |
|---|---|---|---|
| 1 | 需不需要升级存储格式？ | ❌ 不改 | 现有格式有硬伤（不可解析、不可扩展、有隐式 bug） |
| 2 | 需不需要加 API schema/新参数？ | ❌ 不改 | 不改无法达成功能目标，且该变化不影响已有消费者 |
| 3 | 需不需要引入外部依赖？ | ❌ 不加 | 依赖的收益 >= 引入依赖的运维成本 ×2 |
| 4 | 需不需要 LLM 参与决策（自动打分/分类）？ | ❌ 不用 | 规则能完成的场景都不用 LLM；只有语义理解确实绕不开时才用 |
| 5 | 需不需要分层/分级存储（Core/Archive）？ | ❌ 不用 | 注入时截断 + 磁盘全保留等价于但比分层简单 |
| 6 | 需不需要遗忘/淘汰机制？ | ❌ 不主动淘汰 | 磁盘上限兜底 + 用户手工清理就够；除非有法规合规要求 |
| 7 | 需不需要语义检索/embedding？ | ❌ 不用 | FTS5 + 纯文本规则覆盖 95%；剩余 5% 等真的需要了再加 |

### 底线原则

> **做取舍的原则：当一个方案听起来高大上，但想不明白具体能带来多少收益时，默认选更简单的那个。** 等用户确实遇到了那个痛点，再升级。

## 输出产物

做完取舍后，产出中必须包含：

### ✅ 要做的

最终决定做的改动列表，按逻辑分组编号（分组 A / 分组 B）。每个改动写清楚：
- 改什么（代码位置）
- 怎么改（多少行，改哪些方法）
- 为什么（对应取舍清单中的哪个决策）

### ❌ 被否决的方案

按取舍清单编号列出被否决的方案，每条写明：
- 被否决的是什么
- 否决理由（简洁）
- 什么条件下可以再考虑

示例：

| 否决项 | 理由 | 何时可再考虑 |
|---|---|---|
| JSON Lines 存储 | § 分隔 + `\|t:` 前缀已够用，JSON Lines 引入解析复杂度和转义问题 | 当需求扩展到按 tag 快速过滤且条目 > 500 条 |
| LLM 自动打分排序 | 每次写记忆都调一次模型，成本高延迟大；时间倒序已够用 | 当条目超过 100 条且时间排序明显不够用时 |
| Ebbinghaus 遗忘曲线 | 需要跟踪每个条目的年龄 + 被引用次数 + 上次使用，极端过度工程化 | 永不 — 用户手动清理成本远低于遗忘曲线复杂度 |

## 落实方式

1. **加载此 skill** → 过一遍取舍清单
2. **写出取舍分析（上述表格）** → 让用户确认
3. **用户确认后** → 写实现 plan
4. **代码实现** → 按 plan 执行，每次改动跑语法检查
5. **测试验证** → 写内联测试（零 mock 文件 IO），测全部边界场景
6. **确认上线** → 问用户是否要重启/生效
